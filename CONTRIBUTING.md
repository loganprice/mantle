## Getting Started

### Prerequisites

To build and run Mantle locally, you will need:

*   **Go**: 1.25 or later.
*   **Docker / BuildKit**: A running host capable of executing container build commands.
*   **Make**: To execute predefined development scripts.

### Local Development

1.  **Fork and Clone** the repository:
    ```bash
    git clone https://github.com/loganprice/mantle.git
    cd mantle
    ```
2.  **Build** the overarching CLI tool:
    ```bash
    make build
    ```
3.  **Run** the tool to see available commands:
    ```bash
    ./bin/mantle --help
    ```

## Development Standards & AI Rules

This project enforces strict development standards, governed by our AI Agent Rules located in the `.agents/` directory. **Both human contributors and AI agents are expected to adhere to these rules.**

When contributing, please review:
*   **[Uber Go Style Guide](.agents/skills/uber-go-style/SKILL.md)**: We strictly adhere to the Uber Go style conventions (e.g., using table-driven tests, minimizing pointer bounds, standardizing error formatting).
*   **[Linting Standards](.agents/rules/linting.md)**: Your code must pass `golangci-lint` cleanly before submission.
*   **[Testing Standards](.agents/rules/testing-standards.md)**: New features and bug fixes must be accompanied by robust unit tests. Table-driven structure is preferred.
*   **[Documentation Rules](.agents/rules/update-docs.md)**: If your change modifies the configuration schema (`mantle.yaml`) or architecture, you must update the corresponding `docs/`.
*   **[Code Review Strictness](.agents/rules/code-reviewer.md)**: Expect a thorough review focusing on security, architectural integrity, and performance.

### Understanding the Architecture

Mantle uses BuildKit's Low-Level Builder (LLB) to assemble images. If you have never worked with LLB or need a refresher, please read **[Understanding BuildKit LLB](docs/contributing-llb.md)** before attempting to modify graph mechanics in the `internal/assembler` or `internal/pipeline` packages.

## How to Contribute

### Reporting Bugs

Before creating bug reports, please check the existing issues as you might find out that you don't need to create one. When you are creating a bug report, please include as many details as possible:

*   A clear and descriptive title.
*   The exact `mantle.yaml` file (or a minimal reproducible example) that caused the issue.
*   The output/logs generated by the bug.
*   Your OS and Go version.

### Suggesting Enhancements

Enhancement suggestions are tracked as GitHub issues. When creating an enhancement request, please provide:

*   A clear and descriptive title.
*   A detailed description of the proposed functionality.
*   Why this enhancement would be useful to most users.
*   Examples or mockups of how the feature would work (e.g., proposed additions to `mantle.yaml`).

### Pull Request Process

1.  **Create a branch**: Create a new branch for your feature or bug fix (`git checkout -b feature/your-feature-name` or `fix/your-bug-fix`).
2.  **Implement your changes**: Write your code, ensuring you follow the Development Standards outlined above.
3.  **Ensure formatting**: Run `make fmt` to format your Go code imports and syntax.
4.  **Lint your code**: Run `make lint` to catch and resolve any warnings. No PR will be merged with failing lint checks.
5.  **Run tests**: Run `make test` and `make test-cover` to verify nothing is broken and that your new code is well-tested.
6.  **Update documentation**: If applicable, update the `README.md` or files in `docs/` to reflect your changes.
7.  **Commit**: Write clear, concise commit messages.
8.  **Open the PR**: Provide a comprehensive description of what your PR solves, linking to any relevant issues.

## Testing & Linting

We heavily rely on automated verification. You can run all necessary checks locally through `make`:

```bash
# Formats code imports and syntax using gofmt and goimports
make fmt

# Lints the entire project using golangci-lint
make lint

# Runs the standard Go race tests
make test

# Generates a coverage report to highlight untested areas
make test-cover
```

Thank you for contributing to the security and reproducibility of container images!
